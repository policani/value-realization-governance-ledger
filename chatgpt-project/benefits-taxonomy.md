# Benefits Taxonomy

Use this taxonomy to classify approved value claims consistently.

| Benefit type | Typical metric examples | Caution |
|---|---|---|
| Cycle-time reduction | Days from intake to approval, hours per transaction, lead time | Avoid confusing isolated speed with sustainable throughput. |
| Cost avoidance | Avoided vendor spend, avoided manual labor growth, avoided defects | Requires finance confirmation before claiming official value. |
| Risk reduction | Control coverage, audit findings, incident likelihood, exposure reduction | Often needs risk owner validation. |
| Adoption improvement | Active users, process compliance, workflow usage, trained users | Usage is not automatically value. Tie to work outcome. |
| Revenue protection | Retained revenue, reduced leakage, fewer billing errors | Requires business / finance validation. |
| Quality / rework reduction | Defect rate, rework hours, exception count, support tickets | Confirm source and time window. |
| Capacity release | Hours returned, queue reduction, backlog reduction | Distinguish theoretical hours from redeployed capacity. |
| Decision quality | Fewer reversals, better approval cycle, improved evidence completeness | Usually needs proxy evidence and qualitative support. |

## Rule

A benefit can have more than one type, but the ledger should identify one primary type and any secondary types.
